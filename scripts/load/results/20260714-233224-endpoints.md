# A. 처리량 축 (MODE=rps, constant-arrival-rate) — "초당 N건 버티나?"

## A-1. 읽기 3종 (RATE 2000 / 5000 / 8000)

| 엔드포인트 | p95 @2000 | p95 @5000 | p95 @8000 | 판정 |
|-----------|----------:|----------:|----------:|------|
| `GET /chat-rooms` | 1.1ms | 1.9ms | 7.9ms | 8000까지 여유 (0 err, 0 drop) |
| `GET /itineraries` | 1.1ms | 1.4ms | 6.9ms | 8000까지 여유 (0 err, 0 drop) |
| `GET /itineraries/{id}` | 1.6ms | **114.6ms** | **573.7ms** (drop 18,877) | **~5000에서 포화** |

# B. 동시성 축 (MODE=vus, constant-vus) — "동시 접속 N명 버티나?"

VU를 고정해 최대한 밀어붙인다(no-sleep). 처리량은 서버 천장에서 포화하고, **p95는 VU에 비례해 상승**(Little's law).

## B-1. 읽기 3종 (VUS 100 / 400 / 800)

| 엔드포인트 | VUS 100 | VUS 400 | VUS 800 | 처리량 천장 |
|-----------|--------:|--------:|--------:|-----------|
| `GET /chat-rooms` | 11,924rps · p95 10ms | 12,121 · 37.6ms | 11,970 · 76ms | **~12,000 rps** |
| `GET /itineraries` | 13,039rps · p95 9.1ms | 12,761 · 36.7ms | 12,786 · 71.6ms | **~12,800 rps** |
| `GET /itineraries/{id}` | 9,108rps · p95 12.6ms | 9,082 · 49.5ms | 8,573 · 114ms | **~9,000 rps** |

## B-2. 쓰기 2종 (VUS 100 / 400)

| 엔드포인트 | VUS 100 | VUS 400 | 처리량 천장 |
|-----------|--------:|--------:|-----------|
| `POST /chat-rooms` | 4,662rps · p95 24.3ms | 4,614 · 97.5ms | **~4,600 rps** |
| `PATCH /itineraries/{id}/day-plans` | 3,679rps · p95 31.7ms | 3,935 · 112.6ms | **~3,900 rps** |

전 구간 에러율 0%. (VU 모드는 최대 처리량을 밀어내므로, RATE로 상한을 두지 않은 실제 천장이 드러난다.
읽기 목록 ~12–13k, 상세 ~9k, 쓰기 3.9k–4.6k rps.)

> **두 축은 서로 다른 걸 잰다.** 처리량 축(open model)은 도착률을 강제해 **포화 근처의 큐잉 지연**을
> 드러내고, 동시성 축(closed model)은 VU가 응답을 기다렸다
> 다음 요청을 보내므로 **순수 서버 처리 천장과 VU당 지연**을 본다. 숫자를 1:1로 비교하지 말 것.

## 핵심 발견

1. **의외의 병목은 쓰기가 아니라 `GET /itineraries/{id}`(상세 읽기).** ~5000 RPS에서 포화(p95 114ms),
   8000에서 붕괴(p95 573ms + 대량 drop). 상세 조회는 day_plans **JSONB 파싱 + 아이템 시간순 index 재부여**
   연산이 있어, 단순 목록 읽기(8000까지 여유)보다 CPU 비용이 크다.
2. **쓰기(POST 생성, PATCH day-plans)는 3000 RPS까지 0 에러**로 예상보다 튼튼(p95 ≤21ms).
   (이 박스에선 3000 이상도 여력 있어 보이나, POST의 데이터 대량 생성/정리 비용 때문에 상한 추가 탐색은 생략.)
3. 목록 읽기(`GET /chat-rooms`, `GET /itineraries`)는 8000 RPS도 p95 <8ms로 가볍게 처리.

### 우선순위 튜닝 대상
가장 먼저 손봐야 할 곳은 **`GET /itineraries/{id}` 상세 조회의 index 재계산/직렬화 경로**다.
캐싱 또는 index를 저장 시점에 계산해 두면 읽기 천장이 크게 오를 것으로 예상.

## 이번 측정에서 얻은 방법론 교훈 (스크립트에 반영됨)

1. **teardown 오염**: k6는 setup/teardown의 HTTP 요청도 전체 지표에 합산한다. 생성(POST) 엔드포인트는
   정리 DELETE가 수천 건이라 rate/p95가 심하게 왜곡됐다(POST가 677rps로 오측정 → 실제 3000+).
   → 액션 요청만 `name` 태그 **하위지표**로 분리 측정해야 정확.
2. **PATCH "No changes detected"**: `PATCH /itineraries/*`는 직전과 동일 값이면 400 반환.
   → 수정 부하는 반복마다 페이로드를 변화시켜야 함(카운터 적용).
3. **동일 리소스 동시 수정 경합**: 한 일정을 여러 VU가 동시 수정하면 ~50% 400(트랜잭션+스냅샷 충돌).
   → VU마다 다른 리소스를 쓰도록 리소스 풀 + `maxVUs = 풀 크기`(VU↔리소스 1:1)로 측정.
